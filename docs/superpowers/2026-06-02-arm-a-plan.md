# Arm A (production-realism) — implementation plan

The controlled arm (B, built) isolates **weights** by giving both backends the
same thin prompt. **Arm A** answers the *product* question: the **shipped API**
(rich ~1600-token RAG prompt + retrieved few-shot + register/shape directives +
decode nudges) vs the **LoRA** in its thin serving condition — "which deployed
system replies more like Bohdan." This plan makes it a small, *safe* job.

## The one hazard: a fresh leak on the API side

The hold-out turns (`eval_split_for`) **are still in Qdrant/BM25** — only the
*temporal* `eval_split` is excluded from retrieval, not the recipient-stratified
one we score on. So the rich-API retrieval could surface the **gold reply** as a
few-shot example → instant leak, this time favouring the API. Arm A is invalid
unless the scored `turn_id`s are excluded from retrieval.

## Steps

1. **Turn-id-tagged hold-out.** Load from the DB instead of `eval.jsonl`:
   `select PersonaTurnRow where eval_split_for(id) is True` → `(turn_id,
   recipient_id_hash, incoming_context_json, your_reply)`. (Same set as
   `eval.jsonl`, now carrying ids + recipient for per-register breakdown.)
2. **Reuse the LoRA gens** from arm B (`data/eval/compare/main/pairs.jsonl`
   `gen_lora`) if item order is preserved, or regenerate (cheap, local).
3. **API arm = the production path.** Build the rich prompt via
   `persona_rag.generate.prompt.build_messages` with `GENERATION_BACKEND=openai`
   and run the real retrieval — **but exclude the scored ids**. Recommended:
   post-filter `retrieve_hybrid` results, dropping any retrieved turn whose id is
   in the eval set (a contained change; alternative: add an `eval_split_for`
   filter to `qdrant_store` + `bm25_store` mirroring the existing temporal one).
4. **HARD leak assertion.** For every item, assert the gold `turn_id` **and** the
   exact real-reply text are absent from the retrieved set; **fail the run** if
   not. This keeps arm A trustworthy even if the exclusion is imperfect.
5. **Score identically.** `compare_scorecard(real, gen_api_rich, gen_lora_thin,
   train_replies=...)` → report arm A next to arm B in the findings.
6. **Pin + log decode levers** per the audit's fairness table (§4.5): fixed
   `BEST_OF_N=1`; report the API with **and** without the `)`/`!` logit-bias so
   the tic numbers are labelled *steered* vs *learned*.

## Where it lands

- Extend `scripts/compare_persona.py` with an `--arm {controlled,production}`
  flag (production = DB loader + rich prompt + retrieval-exclusion + the
  assertion), or a sibling `scripts/compare_persona_armA.py`. Reuse
  `build_messages` + the retrieval nodes; do **not** duplicate metric logic
  (`persona_rag/eval/compare.py` already has it).
- Output `data/eval/compare/armA/`; render with the existing
  `scripts/plot_comparison.py --name armA`.

## What it will (and won't) tell us

Arm A compares **deployed product vs deployed product** — the API *with* the
machinery built to fix exactly the register gaps arm B exposed. Expect the API
to close most of the length/`!` gap there. The honest decide-narrative reports
**both**: B ("which model is more Bohdan, bare") and A ("which product ships
better"). The blind human panel still overrides both.

**Effort:** ~half a day. **Risk:** low *if* step 4's assertion is in place.
