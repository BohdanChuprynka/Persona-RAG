## Interpretation

1. **Length is the decisive, robust win for the LoRA.** Raw `gpt-4o-mini` sits ~128‚Äì135 chars (earth-mover) away from Bohdan's per-bubble length ‚Äî it writes long, paragraph-style replies. The LoRA is ~1.5‚Äì2.9 away (it matches his terse, multi-bubble texting). The Œî CI excludes 0 in both seeds ‚Üí not noise.
2. **Punctuation register: LoRA matches the "no `!`" rule exactly** (0.00 vs the API's ~0.63). Under the thin prompt the base model reverts to its peppy assistant register.
3. **Message shape is a statistical tie.** `shape_js` favors the LoRA numerically but the Œî CI spans 0 at the n=300 noise floor (~0.02‚Äì0.03) in both seeds ‚Äî call it a tie, not a LoRA win.
4. **Opener variety / diversity:** LoRA opens more variably (higher entropy) but repeats whole replies slightly more (distinct 0.96‚Äì0.99 vs 1.0) ‚Äî mild, watch for mode-collapse at scale.

## Caveats (do not over-read this)

- **Controlled arm only ‚Äî this is NOT the shipped product.** Both backends ran on the *thin* prompt. The *deployed* API path adds the rich ~1600-token RAG prompt, retrieved few-shot, register/shape directives, and the `)`/`!` logit-bias ‚Äî machinery built specifically to fix the exact register gaps measured here. The shipped API would score far better on length and `!`. **This experiment isolates raw model voice (weights), not "which product ships better."** That is **arm A (production-realism)**, still to build (needs a `turn_id`-tagged hold-out + retrieval exclusion ‚Äî see the design doc ¬ß7).
- **LoRA near-copy of training replies is 7‚Äì15% (API 0%).** It exceeds the provisional Œµ‚â§5% guardrail. But short casual replies (`–æ–∫`, `—Ç–∞ –Ω–æ—Ä–º`, `+`) naturally overlap with training text, so this is **not yet evidence of overfitting** ‚Äî it needs a *length-controlled* baseline (how often do *real held-out* replies near-match *train* replies?) before being called memorization. Flagged, not concluded.
- **Automatic metrics are surface proxies.** None of them measure "feels like Bohdan" directly. The **blind human panel is the real verdict** (kit built, below).
- **Single decode per item at temp 0.8.** CIs here are sampling (bootstrap) uncertainty; decode-variance and a greedy stable-anchor run are cheap follow-ups (configurable in the runner).

## The verdict gate ‚Äî blind human panel (built, awaiting Bohdan)

`reports/main/human_eval/rater.html` ‚Äî 100 blind, randomized forced-choice pairs ("which reply is more like something you'd send?"). Score `choices.json` against `key.json` ‚Üí win-rate + Wilson CI. This resolves the actual target the metrics only approximate.

## Provisional recommendation

Under identical minimal prompting, the fine-tuned 3B is **much closer to Bohdan's register** (length, punctuation) than raw `gpt-4o-mini` ‚Äî exactly what the fine-tune was for ‚Äî and runs locally at $0/reply with comparable median latency. But the **ship decision** for the bot needs three things this run does not yet provide: (1) **arm A** (the API *with* its production scaffold), (2) the **blind human win-rate**, and (3) the **copy-rate caveat resolved** with a length-controlled baseline. Treat this as a strong, honest *first* result, not the final verdict.
